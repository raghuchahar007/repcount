# GymRep M3 — Design Document

**Date:** 2026-03-05
**Sprint:** M3 — Platform Hardening + Gym Growth
**Status:** Approved

---

## 1. Public Gym Directory (Landing Page)

**Problem:** `gymrep.in/` currently redirects logged-out users to `/login`. No discovery surface.

**Design:**
- `/` renders a new `LandingPage` component — public gym directory grid
- Each gym card shows: logo, name, city, facilities tags, timing (slots or "24×7")
- Clicking a card → `/gym/:slug` (already exists)
- Logged-in users see "Go to dashboard →" in header instead of Login/Register
- New API: `GET /api/public/gyms` — returns active gyms (name, city, slug, logo_url, facilities, timing_mode, timing_slots)
- `RootRedirect` component updated: logged-in → dashboard, logged-out → `LandingPage`

---

## 2. Gym Name + City Uniqueness + 1 Gym Per Owner

**Problem:** `name` has no uniqueness constraint. `slug` is globally unique but doesn't prevent name squatting. No limit on gyms per owner.

**Design:**
- Add compound unique index: `{ name: 1, city: 1 }` on Gym model (case-insensitive via collation or lowercase normalization)
- On gym create: check if owner already has a gym → return 409 "You already have a gym registered"
- On gym create: check name+city conflict → return 409 "A gym with this name already exists in this city"
- City field: trim + lowercase before storing for consistent comparison

---

## 3. Gym Claim / Dispute Flow (Layer 3)

**Problem:** Someone can register a gym name they don't own. Legitimate owner has no recourse.

**Design:**
- New `GymClaim` model: `{ gym, claimant_name, claimant_phone, claimant_email, reason, status: 'pending'|'approved'|'rejected', created_at }`
- Public gym page (`/gym/:slug`) → small "Claim this gym" link in footer
- Claim form: name, phone, email, reason (free text)
- `POST /api/public/:gymId/claim` — creates claim record
- Super admin panel: `/super-admin/claims` page listing pending claims
- Super admin action: "Transfer ownership" → changes `gym.owner` to a new User by email
- No automation — manual review only at this stage

---

## 4. Plan Types + Pricing

**Problem:** Memberships have a fixed plan_type enum (monthly/quarterly etc.) with no gym-specific plan categories (e.g., Strength vs Strength+Cardio).

**Design:**
- Add to Gym model: `plan_types: [{ id: String, name: String }]` — default `[{id:'strength', name:'Strength'}, {id:'strength_cardio', name:'Strength + Cardio'}]`
- `pricing` restructured: `{ [planTypeId]: { monthly, quarterly, half_yearly, yearly } }`
- Add `plan_type_name: String` to Membership model
- Settings UI: plan type manager (add/rename/remove) + pricing grid (rows = plan types, cols = durations)
- When recording a membership: owner selects plan type first, then duration

---

## 5. Timing Slots

**Problem:** Gym has a single `opening_time`/`closing_time` range. Tier 2/3 gyms commonly have morning + evening sessions.

**Design:**
- Add to Gym model:
  - `timing_mode: 'slots' | '24x7'` (default: `'slots'`)
  - `timing_slots: [{ label: String, open: String, close: String }]` (default: `[{Morning, 06:00, 12:00}, {Evening, 16:00, 22:00}]`)
- Remove `opening_time` / `closing_time` fields (migration: convert existing values to first slot)
- Settings UI: toggle 24×7 mode; slot editor (add/edit/remove slots)
- Displayed on: gym card (public directory), public gym page, member home

---

## 6. Attendance Improvements

**Problem:** (a) Duplicate same-day check-in throws 500 error instead of friendly message. (b) Owner has no "who's here today" view. (c) No busy hours insight.

**Design:**

**Duplicate check-in:**
- Attendance model already has `{ member, gym, check_in_date }` unique index
- QR check-in route: catch 11000 error → return 200 with `{ already_checked_in: true, checked_in_at: <time> }`
- Scanner UI: show "Already checked in today at 9:14 AM" instead of error

**Today's check-ins (owner):**
- New widget on owner dashboard: "Today — N check-ins" with expandable list (member name + time)
- API: `GET /api/gym/:gymId/attendance/today` — members who checked in today

**Busy hours:**
- New analytics section on owner dashboard
- API: `GET /api/gym/:gymId/attendance/busy-hours` — aggregate `checked_in_at` by hour of day (last 30 days)
- UI: simple horizontal bar chart (hour labels + bar width proportional to check-ins)

## 7. Join Gym → Request Only

**Problem:** Members can directly join a gym without owner approval.

**Design:**
- Remove direct `joinGym()` API call from member UI
- `JoinGymPage`: search gym by slug → preview → "Request to Join" button only
- On request: create Lead `{ source: 'app_request', user: userId, status: 'new' }` (backend already supports this)
- Member sees "Request sent — pending owner approval" state with "Cancel Request" option
- `GET /api/me/join-requests` — returns pending app_request leads for current user
- Owner approves/rejects from `/owner/join-requests` (already exists)
- Leads page: filter explicitly excludes `app_request` source to prevent overlap

---

## 8. Auth Fixes

**Problem:** Multiple auth state management bugs across routes.

**Fixes:**
1. `/login` — add redirect to dashboard if `user` is already set
2. `/register` — same redirect guard
3. `/verify` — redirect to `/login` if `location.state?.phone` is missing
4. `/choose-role` — redirect to dashboard if `user.role` is already set
5. Login page: remove phone/OTP tab entirely (hide, not delete — can re-enable later)
6. Registration: phone 11000 error → "This phone number is already registered"
7. Verify `User.phone` sparse unique index is active post `drop-stale-indexes`

---

## 9. Profile Pages

**Member profile (`/m/profile`):**
- Add prominent active membership card at top: plan type name, expiry date, days remaining, status badge
- Current state: membership history is shown but buried

**Owner profile:**
- New page `/owner/profile`: editable name + email, gym name + city (read-only summary), "View public page →" link
- Currently owners only have gym settings, no personal profile page

---

## 10. Settings Stay-After-Save

**Problem:** Settings form may reset after save.

**Fix:**
- Audit `Settings.tsx`: ensure `loadGym()` is only called on mount (not after save)
- After save: update local state from API response directly, do not re-fetch
- Show inline "Saved ✓" confirmation next to save button (fades after 3s)

---

## Data Model Changes Summary

| Model | Change |
|-------|--------|
| Gym | Add `plan_types[]`, restructure `pricing`, add `timing_mode` + `timing_slots[]`, remove `opening_time`/`closing_time`, add compound unique index `{name, city}` |
| Membership | Add `plan_type_name: String` |
| GymClaim | New model |

## Migration Notes
- Existing gyms: set `timing_slots` from `opening_time`/`closing_time`, set `timing_mode: 'slots'`
- Existing memberships: `plan_type_name` can default to `''` or `'Strength'`
- Run `drop-stale-indexes` after Gym model changes
